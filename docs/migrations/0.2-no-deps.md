# Migration: `--no-deps` default flip (v0.1.x → v0.2.0)

**Applies to**: anyone upgrading `runsteps` from v0.1.x to v0.2.0.
**Why this exists**: v0.1.x silently passed `--no-deps` to every `just` invocation, which caused prerequisite-only recipes (e.g. `deploy-all: setup deploy-infra deploy-services`) to be silent no-ops — exit 0, nothing ran. This is dangerous in deployment contexts. v0.2.0 fixes the root cause.

## What changed

### v0.1.x (old behavior)
`runsteps` invokes every `just_recipe` step as:
```
just --no-deps <recipe>
```
This stripped `just`'s own prerequisite system. `runsteps`' `depends_on` was the only path to run prereqs.

### v0.2.0 (new default)
`runsteps` invokes every `just_recipe` step as:
```
just <recipe>
```
`just` now runs its own prereqs unless you explicitly opt out.

### v0.2.0 opt-out (new field)
Add `just_no_deps = true` to any step that needs the v0.1.x isolation:
```toml
[[steps]]
name = "deploy"
just_recipe = "deploy"
just_no_deps = true   # v0.1.x-style: skip just's prereqs
```

## How to migrate

### 1. Audit your `runsteps.toml` for affected steps

For each step with `just_recipe = "..."`, check whether the named recipe has prerequisites in your `justfile`.

Run:
```sh
just --dump --dump-format json --justfile <path> | jq '.recipes[<recipe-name>].dependencies'
```

If the list is non-empty, the recipe has prereqs. Decide per step:
- **Let `just` run the prereqs** (the new default): remove the step's explicit `depends_on` that duplicates those prereqs, or leave everything as-is (idempotent operations will re-run; harmful operations need the next option).
- **Keep v0.1.x isolation** (skip prereqs): add `just_no_deps = true` and keep your `runsteps.depends_on` chain exactly as it is.

### 2. Prefer flattening over `just_no_deps`

Per [ADR-001](../adr/001-standalone-task-runner.md), the recommended pattern is to express dependencies in `runsteps.toml` rather than rely on `just`'s prereq tree. For a `justfile` like:

```just
deploy-all: setup deploy-infra deploy-services
```

Instead of one step with `just_recipe = "deploy-all"`, write three steps:

```toml
[[steps]]
name = "setup"
just_recipe = "setup"

[[steps]]
name = "deploy-infra"
just_recipe = "deploy-infra"
depends_on = ["setup"]

[[steps]]
name = "deploy-services"
just_recipe = "deploy-services"
depends_on = ["deploy-infra"]
```

This gives `runsteps` visibility into each step (picker, dry-run, future parallelism).

### 3. Update and test

```sh
cargo install runsteps
runsteps --dry-run    # inspect the planned invocations
```

Confirm the `just ...` lines in the dry-run output match what you intend.

## Escape hatch: pin to v0.1.x

If you need more time to audit your configs, pin your install to v0.1.x:

### Cargo

```sh
cargo install runsteps --version "~0.1"
```

### Homebrew

If the maintainer publishes a `runsteps@0.1` formula:

```sh
brew install silafood/runsteps/runsteps@0.1
```

If no `@0.1` formula exists yet, use cargo. File an issue requesting the pinned tap if needed.

### Shell installer

```sh
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/silafood/runsteps/releases/download/v0.1.5/runsteps-installer.sh | sh
```

Replace `v0.1.5` with the latest v0.1.x tag from the [releases page](https://github.com/silafood/runsteps/releases).

## FAQ

**Q: Will I get a warning before the flip?**
Yes — v0.1.6 ships a warning at config load for any step pointing at a prerequisite-only `just` recipe. The warning names the upcoming default change.

**Q: Does `just_no_deps = true` work in v0.1.x too?**
No. The field is introduced in v0.2.0. In v0.1.x, `--no-deps` is always passed.

**Q: Can I silence the v0.1.6 warning?**
Yes: `RUNSTEPS_NO_WARNINGS=1 runsteps ...` or export it in your shell.

**Q: What if my `justfile` uses complex prereq chains and flattening is not realistic?**
Set `just_no_deps = true` on the affected steps. The opt-out is permanent and fully supported.
